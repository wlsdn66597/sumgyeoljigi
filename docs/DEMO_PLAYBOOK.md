# 숨결지기 2차 데모 플레이북 — 무엇을, 왜 보여줄 것인가

이 문서는 "센서를 붙이는 데 성공했다"가 아니라 **"이게 실제로 사람의 상태를 읽어낸다"** 를
심사자에게 눈으로 보여주기 위한 각본이다. 데모의 성패는 결국 한 가지로 갈린다 —
**내가 무언가를 하면, 화면이 즉시 그리고 말이 되게 반응하는가.** 숨을 참으면 경보가 뜨고,
가짜 소음에는 속지 않고, 몇 분 만에 그 사람의 정상 호흡 범위를 스스로 학습하는 모습.
그게 보이면 데모는 성공이다.

아래 모든 시연과 수치는 마지막 절(정직성)의 원칙을 따른다. 우리는 "일상 모니터링과 환경 케어"를
증명하는 것이지, 의료 성능을 주장하는 게 아니다.

---

## 1. 먼저, 호흡이 어떻게 감지되는가 (설명용 한 장)

사람이 숨을 쉬면 가슴이 수 밀리미터 오르내린다. 60GHz FMCW 레이더는 반사파의 위상 변화로
이 아주 작은 움직임을 잡아내고, 그 결과를 시간축에 늘어놓으면 규칙적인 파형이 된다. 상태마다
파형의 모양이 다르다는 것이 핵심이다.

```
정상 호흡   ∿∿∿∿∿∿∿∿   분당 12~40회(0.2~0.7Hz), 진폭이 규칙적
무호흡      ∿∿∿______   진폭이 갑자기 무너짐 (가슴이 멈춤)
뒤척임      ∿∿✧✦✧∿∿    고주파의 큰 진폭 버스트 (몸 전체가 움직임)
부재        ..........   반사 신호 자체가 없음
```

우리 소프트웨어(`radar_dsp.py`)가 하는 일이 정확히 이 패턴을 해석하는 것이다. 호흡 대역의
FFT 피크 주파수에 60을 곱하면 분당 호흡수가 나오고, 최근 4초의 신호 세기가 직전보다 크게
꺼지면 무호흡으로 의심하며, 1.5Hz 이상의 고주파 에너지가 평소의 두 배를 넘으면 움직임으로 본다.
실제 MR60BHA2 모듈은 이 계산을 자체적으로 수행해 호흡수를 바로 내보내므로, 우리 DSP는
그 값을 검증하고 뒷받침하는 역할로 남는다. 무호흡 판정만은 모듈이 직접 주지 않기 때문에,
"재실 중인데 호흡수가 몇 초간 거의 0"이라는 조건으로 엣지에서 유도한다(`radar_bridge.py`).

정리하면 **숨 → 가슴 변위 → 파형 → 주파수 분석 → 숫자** 로 이어지는 인과의 사슬이 있고,
데모는 이 사슬의 각 고리를 눈에 보이게 만드는 일이다.

## 2. 라이브 데모 3대장

### 2-1. 레이더 조작-반응 6종 — 증거의 핵심

심사자가 확인해야 할 건 단 하나, **입력을 바꾸면 출력이 즉각적이고 논리적으로 반응한다**는 것이다.
그래서 데모는 "내가 이렇게 하면 → 화면이 이렇게 바뀐다"의 짝으로 구성한다.

| 내가 하는 것 | 화면에 나타나는 반응 | 증명되는 것 |
|---|---|---|
| 성인이 레이더 앞에 앉는다 | 호흡 파형이 분당 12~20회 안정 곡선으로 수렴, 재실 = 있음 | 비접촉 호흡 감지 |
| **10초간 숨을 참는다** | 호흡수가 뚝 떨어지고 몇 초 안에 🔴 무호흡 의심 경보 | 이상 감지와 반응 지연 |
| 몸을 흔든다 | 움직임 수치가 튀고 수면 상태가 "뒤척임"으로 전환 | 상태 분류 |
| 아기 울음 소리를 튼다 | 울음 감지 + 다른 신호와 교차검증되어 주의/경보 | 음향 AI |
| 자리에서 벗어난다 | 재실 = 없음 | 재실 판별 |
| (SCD41 연결 후) 센서에 입김을 분다 | CO₂·습도가 오르고 환경 권고가 자동 생성 | 선제 케어 |

여기에 **시간의 흐름을 보여주는 세 가지**를 얹으면 신뢰가 완성된다. 첫째, 호흡 파형은 지금 값
하나가 아니라 최근 60초의 연속 곡선으로 그려지기 때문에 조작 불가능한 라이브라는 게 드러난다.
둘째, 몇 분 지켜보면 "이 사람의 정상 범위는 14~19회/분"이라는 문구가 화면에 뜬다 — 개인화 학습이
실시간으로 일어나고 있다는 증거다. 셋째, 이벤트 로그에 언제 무엇이 왜 경보됐는지 타임스탬프로
남는다.

### 2-2. 속지 않는 데모 — 우리의 진짜 차별점

사실 제일 강한 카드는 호흡 감지 그 자체가 아니라 **융합이 오탐을 죽인다**는 점이다. 단일 센서
제품과 갈라지는 지점이 바로 여기이기 때문이다. 소프트웨어 검증에서 우리는 "단일 센서 오탐률
57.6% → 융합 2.7%"를 보였는데, 이 장면을 실물로 재연하는 것이다.

시연은 이렇게 흐른다. 먼저 마이크에 **울음이 아닌 소음**(박수 소리나 TV)을 들려준다. 단일 센서
논리라면 여기서 경보가 났을 상황이지만, 레이더가 "호흡 안정, 움직임 없음"이라고 말해주기 때문에
**융합은 침묵한다.** 그런 다음 **진짜 울음과 뒤척임을 동시에** 만들면, 그때 비로소 경보가 올라간다.
덤으로 1초짜리 신호 튐을 만들어 디바운스가 그것을 무시하고 지속적인 이상만 경보하는 것도 보여줄 수
있다.

핵심 메시지는 이것이다 — **"되는 것"만큼이나 "안 되어야 할 상황에서 조용한 것"이 신뢰의 증거다.**
같은 맥락으로 선풍기 바람에 호흡을 오인하지 않는 것, 빈 방에서 재실을 오탐하지 않는 것도 함께
보여줄 수 있다.

### 2-3. 환경 폐루프 — 감지에서 회복까지 한 바퀴

세 번째는 "감지 → 권고 → 개입 → 회복"이 실제로 한 바퀴 도는 모습이다.

방을 건조하게 만들면 SCD41이 습도가 35% 아래로 떨어진 걸 감지하고, 대시보드에 "습도 낮음 →
가습기 동작" 권고가 자동으로 뜬다. 보호자가 확인 버튼을 누르면(초기에는 안전을 위해 반자동)
스마트플러그로 가습기가 켜지고, 습도가 올라가는 곡선이 실시간으로 그려진다. 40%대에 들어서면
권고는 스스로 사라진다.

CO₂ 버전은 더 극적이라 오히려 이쪽을 추천한다. 방문을 닫고 두 사람이 들어가면 CO₂가
1000ppm을 넘겨 차오르는 곡선이 보이고, 환기 권고가 뜬 뒤 창문을 열면 수치가 뚝 떨어진다.
눈에 보이는 인과라 임팩트가 크다. 측정할 값은 감지에서 권고까지의 지연, 개입 후 회복 시간,
그리고 권고가 제때 사라지는 정확성이다.

## 3. 시스템 완성도 데모 (짧지만 임팩트가 크다)

베이비 모니터는 결국 "믿고 맡길 수 있는 물건"이어야 한다. 짧게 세 가지를 보여주면 그 인상을 준다.

- **장애 복구.** 시연 도중 일부러 프로세스를 죽여 보인다. systemd가 3초 안에 자동으로 되살려
  모니터링 공백을 최소화한다. "죽으면 안 되는 물건"이라는 제품 성격에 딱 맞는 장면이다.
- **프라이버시.** MQTT로 오가는 실제 데이터를 화면에 띄운다. 숫자만 흐르고 영상이나 음성 원본은
  저장되지 않는다는 것을 눈으로 보여준다. 비영상 포지셔닝의 시각적 증거다.
- **아침 리포트.** 실제로 하룻밤 쌓은 데이터로 만든 수면 타임라인 그림과 "3시경 자주 깼다" 같은
  인사이트. 부모가 최종적으로 받아보는 산출물이라, 이걸로 이야기가 깔끔하게 마무리된다.

## 4. 실험 세 가지 (보고서에 들어갈 정량 근거)

라이브 데모가 "즉각 반응"을 보여준다면, 아래 실험들은 숫자로 뒷받침한다.

### 실험 A — 환경 폐루프 정량화 (당일 가능)

2-3의 폐루프를 여러 번 반복하면서 감지 지연, 회복 시간, 권고 정확성을 표로 정리한다. 데모 장면을
그대로 측정값으로 바꾸는 것이라 부담이 적다.

### 실험 B — 울음과 환경의 상관을 실물로 재현 (인형 모의환경)

소프트웨어 검증에서 "울음이 있을 때 평균 습도 30%, 평소 37%"라는 상관을 합성으로 보였다. 이걸
실제 센서로 재현하는 통제 실험이다. 인형과 울음을 재생할 스피커, 가습기로 모의환경을 꾸미고 몇
시간 돌린다. 건조한 구간(습도 35% 미만)에서는 울음 클립을 자주(시간당 네 번쯤), 습한 구간
(45% 이상)에서는 드물게(시간당 한 번) 재생한다.

이 실험의 묘미는 **우리가 상관을 일부러 심어두고, 시스템은 그걸 모르는 채로 스스로 발견하는지**를
보는 것이다. 상관 발견 엔진(`cry_context.py`)이 로그만 보고 "울음이 있을 때 습도가 유의하게 낮다"를
산출해내면 성공이다. 결과는 합성 검증치와 같은 형식의 표로 나란히 배치한다. 다만 보고서에는
"의도적으로 주입한 상관을 회수하는 실험이며 실제 영유아의 인과를 증명하지 않는다"고 반드시 밝힌다.

### 실험 C — 여러 밤에 걸친 환경과 수면질의 관계 ("최적 환경"의 정직한 버전)

하룻밤 실험에 조건 하나만 얹으면 된다. 첫째·셋째 밤은 가습기를 끄고(건조), 둘째·넷째 밤은
켜서(40~50%) 밤마다 뒤척임 횟수, 각성 횟수, 안정 수면 비율을 비교한다. 그러면 "이 사람은 습도
45% 부근에서 뒤척임이 적었다" 같은 개인 맞춤 인사이트가 나온다.

이것이 바로 "최적 환경을 찾아준다"를 **정직하게** 보여주는 방법이다. 표본이 작으니 일반화된
결론을 주장하는 대신, "개인 맞춤 인사이트를 산출하는 파이프라인을 시연한다"로 포지셔닝한다. 교란
요인은 같은 침구, 비슷한 취침 시각, 카페인·음주 배제 정도만 메모해두면 충분하다.

## 5. 하룻밤 실험은 이렇게 진행한다

하룻밤 실험은 낮 데모와 목적이 다르다. **사람이 지켜보지 않는 8시간 동안에도 말이 되는 데이터가
안정적으로 쌓이는가**를 증명하는 것이다.

먼저 무엇을 주장할 수 있고 없는지를 분명히 해두자. 증명할 수 있는 것은 8시간 무중단 수집(안정성),
수면 중 호흡수가 성인의 타당한 범위(분당 10~20회)로 모이는지, 움직임 이벤트가 실제 뒤척임과
맞는지, **밤새 오경보가 몇 번 울렸는지**, 그리고 아침 리포트가 자동으로 만들어지는지다. 반대로
의료적인 무호흡 탐지 성능, 영유아 성능, 수면 단계(REM 등)의 정확도는 이 실험으로 주장할 수 없다.

**준비.** 레이더는 침대 옆에 가슴 높이로, 0.5~1.5m 거리에서 몸통을 향하게 두고 밤새 움직이지
않도록 단단히 고정한다(중간에 틀어지면 데이터 전체가 애매해진다). 파이와 레이더 모두 콘센트에
직결하고, 방에는 한 사람만 있어야 한다. 소프트웨어 쪽은 `nuni-recorder`(모든 토픽을 원본
그대로 기록)가 돌고 있는지, 디스크 여유가 있는지 확인하고, 야간에는 알림음을 꺼서 로그만 남긴다.

**진행 순서.** 취침 30분 전에 짧은 캘리브레이션을 한다 — 안정 착석 상태로 30초씩 세 번 수동
호흡 카운트(그날의 정확도 기준점), 숨참기 한 번(시스템이 살아있는지 확인), 1분 자리 비움(재실
감지 확인). 그다음 취침·기상 시각을 적고, 밤중에 깨면 시각만 폰에 메모한다. **이 수면일지가
정답지(ground truth)** 역할을 한다. 스마트워치나 수면 앱을 매트리스에 같이 두면 각성 이벤트를
대조하기 훨씬 좋다. 아침에는 명령 한 줄이면 리포트가 나온다:

```bash
python analyze_night.py ~/nuni_data/nuni_<날짜>.jsonl
```

이 스크립트가 일곱 가지 지표를 자동으로 뽑는다 — 수집 커버리지, 재실 비율과 이탈 구간, 수면 중
호흡수 분포, 움직임 에피소드(수면일지와 대조), **밤새 경보 횟수**, 수면 상태 비율, 개인화 수렴.
여기에 호흡 타임라인 그림도 함께 저장된다.

**미리 알아둘 함정.** 두꺼운 이불은 가슴 변위를 약하게 만든다. 첫 밤은 평소대로 자보고 신호가
약하면 각도나 거리를 조정하는데, 이 과정 자체도 "발견"으로 기록하면 된다. 모로 눕거나 등을 돌리면
신호가 약해지는 구간이 생기는데, 이건 오히려 "자세별 신호 품질"이라는 분석거리가 된다. WiFi가
잠깐 끊기면 공백이 생기는데, 그 공백을 숨기지 말고 그대로 기록에 남긴다. 그리고 **본 실험 전에
1~2시간 낮잠으로 반드시 리허설**을 해서 전체 파이프라인을 한 번 돌려보고 문제를 잡는다. 첫 밤을
바로 본 실험으로 쓰지 않는다.

하룻밤은 어디까지나 파일럿이다. 진짜 가치는 3~5박에서 나오는데, `sleep_rhythm.py`의 "자주
깨는 시간대, 안정 수면 비율" 인사이트가 여러 밤의 데이터를 먹기 때문이다. 파일럿이 성공하면 같은
세팅으로 며칠 반복만 하면 되고(추가 작업은 거의 없다), 합성 검증(5박 기준 각성 14회·새벽 3시경·
안정 수면 87%)과 같은 형식의 실측 표를 만들어 나란히 비교할 수 있다.

## 6. 보고서에 넣을 정량표 체크리스트

- [ ] 호흡수 정확도 — 성인 3~5명에게 수동 카운트와 레이더 추정을 비교해 실측 MAE (합성 0.5회/분과 나란히)
- [ ] 무호흡 감지 지연 — 숨참기 시작부터 경보까지 (로그 타임스탬프)
- [ ] 울음 감지 혼동행렬 — 울음 재생 대 무음/생활소음 각각 여러 번
- [ ] E2E 지연 실측 — 조작부터 화면 반영까지 (기존 ~4초와 비교)
- [ ] 밤새 오경보 횟수 — 1박 이상
- [ ] 환경 폐루프 — 감지 지연과 회복 시간 (실험 A)
- [ ] 울음-맥락 실측 상관 (실험 B)
- [ ] 다박 환경-수면질 비교 (실험 C)
- [ ] 대시보드 스크린샷 6종 + 아침 리포트 산출물

## 7. 정직성 — 발표와 보고서에서 지킬 선

- 모든 실측은 **성인을 대상으로 한 기능 검증**과 **인형 기반 모의환경**에서 이루어진 것이며, 실제
  영유아나 임상 환경의 성능이 아니다.
- 무호흡 감지는 어디까지나 안전 백업 기능이며, 의료기기나 진단을 표방하지 않는다.
- 합성 검증치와 실측치는 언제나 구분해서 표기한다.

## 8. 무엇부터 보여줄 것인가

데모의 3대장은 **레이더 조작-반응 6종(2-1) + 속지 않는 데모(2-2) + 아침 리포트**다. 이 세 개면
"반응한다 · 안 속는다 · 하룻밤을 버틴다"가 한 번에 전달된다. 환경 폐루프(2-3)는 SCD41이 도착한
뒤에 추가하고, 실험 B와 C는 일정에 여유가 있을 때 보고서용으로 돌린다.

그리고 모든 데모는 sim 폴백(가상 센서로 되돌아가는 스위치)을 준비한 상태에서 진행한다. 현장에서
하드웨어가 말썽을 부려도 시연이 멈추지 않도록 하기 위해서다.
