# Phase C — 실물 센서 연동 수행 계획

## 이 단계의 전략

Phase C는 새로운 기능을 만드는 단계가 아니다. 우리는 이미 판단·학습·표시 계층을 전부 완성해서
파이 위에 올려뒀고, 지금 그것들은 **가상 센서**가 만들어내는 데이터를 먹으며 잘 돌고 있다. Phase
C가 하는 일은 그 가상 발행자를 하나씩 **실물 센서로 갈아끼우는 것**뿐이다.

이게 가능한 이유는 모든 모듈이 MQTT 토픽으로만 대화하기 때문이다. 예를 들어 융합 엔진은
`sensor/radar`라는 토픽에서 호흡수·재실·움직임을 받을 뿐, 그 값이 시뮬레이션에서 왔는지 진짜
60GHz 레이더에서 왔는지 알지도 못하고 알 필요도 없다. 그래서 발행하는 쪽만 바꾸면 융합·대시보드·
리포트는 코드 한 줄 고치지 않아도 그대로 동작한다.

작업은 센서를 **한 번에 하나씩** 교체하며 각각 따로 검증한다. 세 개를 동시에 붙였다가 뭐가 문제인지
모르게 되는 상황을 피하기 위해서다. 그리고 실물로 바꾼 뒤에도 가상 센서는 폴백으로 남겨둔다 —
현장에서 하드웨어가 죽어도 데모가 멈추지 않도록.

---

## 시작 전 상태 (Phase B에서 갖춰둔 것)

파이(라즈베리파이 4, 4GB, Debian 13)에는 파이썬 가상환경과 Mosquitto 브로커가 있고, 세 개의
systemd 서비스가 자동으로 돌고 있다. `nuni-edge`는 센서·울음·융합을 상시 구동하는 두뇌이고,
`nuni-dashboard`는 8501 포트에서 대시보드를 띄우는 뷰어이며, `nuni-recorder`는 모든 토픽을
원본 그대로 JSONL로 기록해 야간 실험의 데이터를 남긴다. 접속은 Tailscale(`100.80.124.118`,
사용자 `sooin`)로 무비밀번호 SSH가 되어 있어서 **소프트웨어 작업은 전부 원격으로** 처리한다.
브로커는 ESP32가 붙을 수 있도록 LAN에 열어뒀다(C1의 선행 조건).

가상과 실물 사이의 전환은 `nuni-edge`의 환경변수로 조절한다. `NUNI_SIM_RADAR=0`을 주면 가상
레이더 발행이 꺼지고(실물 레이더 브리지를 켤 때), `NUNI_SIM_ENV=0`을 주면 가상 환경 발행이
꺼진다(SCD41 리더를 켤 때).

역할은 계속 같다. **부품 주문·수령, USB 연결·배선·거치·전원 같은 물리 작업은 사람**이 하고,
**설정·코드·검증 같은 소프트웨어 작업은 원격으로** 진행한다.

## C1. 레이더 (MR60BHA2) — 최우선

**목표:** 실제 호흡·재실·움직임이 `sensor/radar` 토픽으로 흘러 대시보드에 나타나게 한다.

데모의 심장이자 리스크가 가장 큰 부분이라 여기부터 시작한다. 순서는 이렇다.

먼저 **네트워크를 정해야 한다.** XIAO(레이더의 WiFi 두뇌)와 파이가 같은 네트워크에 있어야 서로
통신한다. 데모 이동성을 생각하면 두 가지 선택지가 있다 — 간단하게는 폰 핫스팟에 둘 다 물리는
것이고, 좀 더 멋있게는 파이를 AP(공유기)로 만들어 발표장 와이파이와 무관하게 자급자족하는 것이다.

다음은 **펌웨어에 MQTT를 추가하는 일**이다(USB 연결은 사람, 설정과 플래시는 원격). 키트에 기본
탑재된 ESPHome 펌웨어는 Home Assistant와 통신하도록 되어 있으므로, 여기에 `mqtt: {broker:
<파이 IP>}` 설정을 넣어 다시 플래시하면 상태값을 우리 브로커로 보내기 시작한다. 펌웨어를 처음부터
새로 짜지 않고 기존 것을 그대로 쓰는 이유는, 그게 훨씬 안전할 뿐 아니라 **무호흡 유도 로직이
파이썬 쪽(`radar_bridge.py`)에 있어서 원격으로 얼마든지 튜닝할 수 있기 때문**이다.

펌웨어가 올라오면 **실제 토픽 이름을 확인**한다. `mosquitto_sub -t '#' -v`로 흘러오는 토픽을
보고, `radar_bridge.py`의 `TOPICS` 매핑을 실제 이름(호흡수·재실·움직임·조도)에 맞춰 고친다.
그런 다음 **브리지 서비스(`nuni-radar-bridge`)를 켜면**, ESPHome이 보내는 상태값을 우리 표준
`sensor/radar` 스키마로 1초마다 변환해 재발행한다. 무호흡은 브리지가 유도하는데, "재실 중인데
호흡수가 6 미만인 상태가 4초 넘게 이어지면 무호흡 의심"으로 판단한다(이 임계값들은 환경변수로
바꿀 수 있다). 마지막으로 `nuni-edge`에 `NUNI_SIM_RADAR=0`을 주고 재시작해 가상 레이더를 끈다.

**이 단계가 끝났다고 보려면** 다음이 확인되어야 한다.

- [ ] 앉아 있을 때 대시보드 호흡수가 수동 카운트와 ±2회/분 안에서 맞는다
- [ ] 숨을 참으면 무호흡 경보가 뜬다 (지연 시간을 기록)
- [ ] 자리를 비우면 재실이 없음으로, 돌아오면 다시 잡힌다
- [ ] 몸을 흔들면 움직임에 반영된다
- [ ] `analyze_night.py`용으로 30분짜리 파일럿 수집이 정상적으로 된다

## C2. 울음 (USB 마이크) — 레이더 배송을 기다리는 동안 먼저 해도 됨

**목표:** 실제 소리가 YAMNet 추론을 거쳐 `audio/cry` 토픽으로 나가게 한다.

마이크는 쿠팡에서 오니까 직구하는 레이더보다 먼저 도착할 가능성이 높다. 그래서 순서를 C2 → C1으로
바꿔도 무방하다.

사람이 USB 마이크를 파이에 꽂으면, 원격에서 `arecord -l`로 인식을 확인하고 테스트 녹음을 해본다.
그다음 울음 모델을 준비하는데, 여기에 한 가지 손질이 필요하다. 학습된 모델 파일(`cry_model/
artifacts`의 헤드와 라벨)을 파이로 옮기는 건 간단하지만, **라즈베리파이 4에서 풀 텐서플로우는
너무 무겁다.** 그래서 YAMNet과 헤드를 tflite로 경량 변환하고 `tflite_runtime`으로 추론하도록
`cry_model/infer.py`의 로딩 경로를 바꾼다(마이크 입력을 위한 sounddevice와 portaudio도 설치).
준비가 되면 `cry_classifier.REAL=True`로 전환하고 `nuni-edge`를 재시작한다.

**완료 기준:**

- [ ] 아기 울음 클립을 재생하면 `audio/cry`의 is_crying이 true로 (여러 번 중 몇 번)
- [ ] 무음·일상 대화·TV 소리에는 false로 (오탐 측정)
- [ ] 융합이 제대로 반응한다 — 울음만 있으면 주의, 울음과 뒤척임이 겹치면 경보 (속지 않는 데모 각본)

## C3. 환경 (SCD41) — 도착하면 끼워넣기

**목표:** 실측 CO₂와 온습도가 `sensor/env` 토픽으로 흐르게 한다.

사람이 4핀을 배선한다 — VCC는 3.3V(1번 핀), GND는 6번 핀, SDA는 GPIO2(3번 핀), SCL은
GPIO3(5번 핀). 반드시 파이를 끈 상태에서 연결한다. I2C 자체는 이미 활성화해뒀으므로(재부팅
후 `/dev/i2c-1` 생성됨) 추가 설정은 필요 없다.

이후는 원격 작업이다. `i2cdetect -y 1`로 센서가 **0x62** 주소에 잡히는지 확인하고, 아직 코드가
없는 유일한 부분인 `scd41_reader.py`를 작성한다(sensirion 라이브러리를 쓰거나 smbus2로 직접
읽는다). 이 스크립트가 I2C에서 값을 읽어 `sensor/env`로 발행하며, 조도(lux)는 레이더 쪽에서
받은 값을 브리지가 함께 넘겨준다. 서비스로 등록한 뒤 `nuni-edge`에 `NUNI_SIM_ENV=0`을 주고
재시작하면 실물 환경으로 전환된다.

**완료 기준:**

- [ ] 입김을 불면 CO₂가 급등하고, 환기하면 내려간다
- [ ] 가습기를 켜면 습도가 오르고, 임계값을 넘는 순간 권고가 생겼다가 사라진다 (폐루프 데모 리허설)

## C4. 음성과 스피커

**목표:** 제한된 문장을 알아듣고(`audio/voice` 인텐트) 스피커로 응답하게 한다.

원격에서 Vosk 한국어 소형 모델을 설치하고, 지금 키워드 기반인 `voice_intent.py`를 실제 STT로
바꾼다. 사람이 스피커를 연결하면(파이의 3.5mm나 USB, 또는 노트북) 원격에서 응답을 재생하도록
한다 — 간단한 TTS든 미리 녹음한 wav든 상관없다. 이렇게 하면 데모에 상호작용이 더해져 이야기가
완성된다.

**완료 기준:** 기존 11개 문장 세트를 실제 음성으로 읽었을 때 인식률을 기록한다.

## C5. 데모 패키징과 측정, 그리고 보고서

마지막은 지금까지 붙인 것들을 하나의 데모로 묶고, 숫자로 정리해 보고서에 넣는 일이다.

먼저 **데모를 리허설**한다. DEMO_PLAYBOOK의 3대장과 네 가지 시나리오(성인 비접촉 호흡, 안정
수면, 울음-반복 경보 억제, 환경 권고)를 실제로 돌려보고, 현장 고장에 대비해 sim 폴백으로 되돌리는
절차를 몸에 익힌다. **야간 실험**은 낮잠 리허설로 한 번 점검한 뒤 1박 파일럿을 돌리고, 여유가
있으면 3~5박과 실험 B·C까지 확장한다.

그다음 **측정**이다. DEMO_PLAYBOOK의 정량표 체크리스트를 따라 호흡 MAE, 무호흡 지연, 울음
혼동행렬, E2E 지연, 밤새 오경보 수, 스크린샷 6종을 채운다. 마지막으로 이 실측치들을 **합성
검증치와 구분해서 나란히** 보고서와 README에 반영한다(`NUNI_보고서_반영계획.md`가 했던 방식으로
docx에 소절을 추가). 커밋은 작성자를 wlsdn66597로 하고 Claude 표기는 넣지 않는다.

## 부록 B — 자주 쓰는 명령

```bash
# 서비스 상태 확인 / 재시작 / 로그 보기
systemctl status nuni-edge nuni-dashboard nuni-recorder
sudo systemctl restart nuni-edge
sudo journalctl -u nuni-edge -f

# 실물로 전환하기 (레이더 예시)
sudo systemctl enable --now nuni-radar-bridge
#  → nuni-edge.service 에 Environment=NUNI_SIM_RADAR=0 을 추가한 뒤
sudo systemctl daemon-reload && sudo systemctl restart nuni-edge

# 흐르는 토픽 실시간으로 보기 / 야간 데이터 분석
mosquitto_sub -t 'sensor/#' -t 'fusion/#' -v
.venv/bin/python analyze_night.py ~/nuni_data/nuni_$(date +%Y%m%d).jsonl
```
